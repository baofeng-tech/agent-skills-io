# Release Generation Rules And Runbook

## New Project Capability

This repo now has first-class release generation for three downstream targets:

- `claude-release/`
- `claude-marketplace/`
- `hermes-release/`

The mother skills remain in `targetSkills/`. Release layers are generated outputs, not editing targets.

## Generation Script Entry Points

Primary generation scripts:

- [scripts/build_claude_release.py](/mnt/d/workplace/agent-skills-io/scripts/build_claude_release.py:1)
- [scripts/build_claude_marketplace.py](/mnt/d/workplace/agent-skills-io/scripts/build_claude_marketplace.py:1)
- [scripts/build_hermes_release.py](/mnt/d/workplace/agent-skills-io/scripts/build_hermes_release.py:1)
- [scripts/test_release_layers.py](/mnt/d/workplace/agent-skills-io/scripts/test_release_layers.py:1)

Recommended commands:

```bash
python3 scripts/build_claude_release.py
python3 scripts/build_claude_marketplace.py
python3 scripts/build_hermes_release.py
python3 scripts/test_release_layers.py
```

When real runtime validation needs Python 3.12:

```bash
python3 scripts/test_release_layers.py
```

## Core Rule

Always preserve `targetSkills/` as the cross-platform source layer.

When packaging for a specific platform:

- generate a release layer
- trim non-runtime files
- keep runtime functionality intact where possible
- record residual risk instead of silently hiding it

## Claude Release Rules

Generated by:

- [scripts/build_claude_release.py](/mnt/d/workplace/agent-skills-io/scripts/build_claude_release.py:1)

Rules:

- root-level skill directories for GitHub / skills.sh discovery
- normalize `SKILL.md` frontmatter to Claude-friendly fields
- keep `name`, `description`, `allowed-tools`, optional `when_to_use`
- strip OpenClaw-heavy frontmatter noise
- remove non-runtime files such as `__pycache__`, `*.pyc`, compare scripts, sync scripts, and test helpers
- downgrade risky defaults such as browser auto-open and home-directory persistence when feasible

## Claude Marketplace Rules

Generated by:

- [scripts/build_claude_marketplace.py](/mnt/d/workplace/agent-skills-io/scripts/build_claude_marketplace.py:1)

Rules:

- wrap each generated Claude skill as its own plugin directory
- create per-plugin `.claude-plugin/plugin.json`
- create root `.claude-plugin/marketplace.json`
- use relative `./plugins/<name>` marketplace sources
- keep plugin wrappers thin; runtime content still comes from the generated skill directory

## Hermes Release Rules

Generated by:

- [scripts/build_hermes_release.py](/mnt/d/workplace/agent-skills-io/scripts/build_hermes_release.py:1)

Rules:

- organize skills by Hermes-style category directory
- keep `SKILL.md + scripts + references` structure
- add `metadata.hermes.tags`
- add `metadata.hermes.related_skills` when skill families clearly exist
- add `required_environment_variables` when an API-backed skill needs secrets
- preserve relative paths and progressive disclosure
- trim non-runtime files and risky defaults the same way as Claude packaging

## Security / Suspicious Reduction Rules

Across all release layers:

- remove generated caches and compiled artifacts
- remove compare/sync/test helper shell scripts from published bundles
- prefer repo-local persistence over user-home defaults
- avoid browser auto-open by default in release copies
- avoid documentation that implies cookie extraction or credential scraping

## Real Testing Rules

Release tests must use:

- real credentials from [example/accounts](/mnt/d/workplace/agent-skills-io/example/accounts:1)
- real upstream API responses
- real platform-oriented output directories

The current test harness is:

- [scripts/test_release_layers.py](/mnt/d/workplace/agent-skills-io/scripts/test_release_layers.py:1)

Testing model:

1. all generated skills get structure validation
2. representative unique implementations get real-data smoke tests
3. high-risk write flows are tested separately and explicitly recorded

## Output-Contract Hardening Rule

For LLM-backed CLI skills that expose `--output json`:

- instruct the model to return JSON only, with no prose or markdown fences
- use provider-level JSON response hints when available
- keep parsers tolerant of fenced JSON, bare JSON objects, and balanced trailing objects from live responses
- treat output-contract failures as real runtime bugs, not as acceptable test noise

Reference example:

- [targetSkills/stock-analysis/scripts/analyze_stock.py](/mnt/d/workplace/agent-skills-io/targetSkills/stock-analysis/scripts/analyze_stock.py:1)

## Recommended Execution Order

1. Generate `claude-release/`
2. Generate `claude-marketplace/`
3. Generate `hermes-release/`
4. Run structure + smoke validation
5. Write/update reports
6. Push the release layer to its target repo

Important sequencing note:

- do not run `build_claude_release.py` and `build_claude_marketplace.py` in parallel
- `build_claude_marketplace.py` reads `claude-release/` as input and can under-generate if the release tree is still rebuilding

Concrete command sequence:

```bash
python3 scripts/build_claude_release.py
python3 scripts/build_claude_marketplace.py
python3 scripts/build_hermes_release.py
python3 scripts/test_release_layers.py
```
